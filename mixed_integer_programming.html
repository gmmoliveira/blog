Solving Mixed Integer Programming

Introduction

This article proposes a brief introduction to Linear Programming (LP) in addition to a practical software implementation written in python to solve a formulation called Mixed Integer Programming (MIP) where each entry of X* may be arbitrarily chosen to be real-valued or integer-valued. Here, X* refers to an array containing the optimal solution to a MIP problem, where being optimal means that the numbers in X* results in the minimum/maximum possible objective value, under a set of constraints.

This article is focused on the MIP formulation since it's the most generic one, it's powerful enough to solve very complex real-word problems, even though it's based on a set of linear relationships only (non-linearity is not allowed here). Furthermore, the python implementation presented here uses Google OR-Tools (Operations Research Tools) which is implemented in C++ and it's computationally very efficient. Inside the OR-Tools there are multiple LP solvers, the one used in the scope of this work is called Glop.

Linear Programming Mathematical Formulation

LP is an optimization science studied under the Operations Research (OR) field and it refers to arranging a plan (programming) to solve a problem based on a set of linear relationships. A LP problem contains an objective, which is either maximization or minimization, an objective function determining how the profit/cost changes in a (potentially) high dimensional real-valued space. It's important to understand that the objective function might be formulated to represent any kind of gains, which we would be interested in maximizing, or it could be seen as any kind of loss, which we would be interested in minimizing. Such optimization occurs under constraints of linearly related equations (inequalities may be converted to equalities by adding slack variables, that's automatically done by the proposed framework, however, it may also be delegated to the user) and the feasible solutions are also constrained to be bounded inside certain ranges and certain sets of numbers (such as the set of non-negative integers). Furthermore, X is a vector defined as X = {x0, x1, ..., xn} which represents the solution to the problem. During the process of actually solving the problem, many different answers are visited, however, when the algorithm finishes that answer will be optimum, therefore, denoted by X*, that is, if at least one feasible solution exists. If it's a scenario of alternate optima, i. e., multiple vectors X* exists yielding optimum solutions, Glop will stop once finding the first one. A is a 2-dimensional matrix of k rows and n columns, representing the constraints of the model, it limits the relationship between variables. Geometrically, it defines a subspace in a potentially high dimensional real-valued space into which the solution is meant to be searched. C is a vector defined as C = {c0, c1, ..., cn} representing the objective function, i. e., the goal to either maximize or minimize. While capital letter C represents the objective function, lower case c* represents the optimum objective value, i. e., the minimum/maximum possible objective value, given by the product between X* and C. Vector b has length k and represents the bounds for each constraint in A. Finally, lj and uj are the bounds on the possible values Xj may assume j {0, 1, ..., n}, which by default are [0, +∞). Note that the lower bound of any Xj may never assume negative values, due to matters of mathematical consistency. From this point and forward, the actual variable names will be slightly adapted to match a more concise formulation for a software application, therefore, lj and uj will be mapped by a variable, yet to be introduced, called vars_properties and the bounds b on the constraints will be represented as lower and upper bounds by lb and ub, respectively.

Solving LP problems using Google OR-Tools

This section is concerned on defining software routines which facilitates the usage of solvers inside the framework OR-Tools. Let's start by importing the necessary package(s): 
from ortools.linear_solver import pywraplp
import numpy as np
Next let's create a convenience function which is meant to return the library's internal representation of infinity, it should be used only as bounds for either variables or constraints when needed, using this value elsewhere doesn't make sense. Furthermore, negative infinity may be achieved by multiplying infinity by -1. 
def infinity():
	'''
	output:
	------
		* returns the solver's infinity value
	'''
	return pywraplp.Solver_Infinity()
It's time to create the wrapper function to facilitate the usage of the solver. This function solves a given linear programming (LP) of the following type: 

max/min 	Cx
s.t.		lb <= Ax <= ub
		vars_properties[i][j] <= x <= vars_properties[i][j+1]
		x belongs to {binary, integer, continuous}
It's time to elaborate what should be the inputs and outputs for the convenience function which is yet to be created, the following definitions should be taken as a documentation for such function:

Inputs

  • C: the objective function to minimize or maximize as a 1D numpy array of n elements, one for each variable;
  • A: the constraints coefficients as a 2D numpy array of k constraints by n variables. Slack variables are not needed to be handled;
  • lb: the lower bounds for each constraint as a 1D numpy array of k elements;
  • ub: the upper bounds for each constraint as a 1D numpy array of k elements;
  • vars_properties: a dictionary keyed by indexes of decision variables which defines the properties of each var as explained below:
    • if method is BOP, vars_properties won't be considered;
    • if method is CBC:
      • vars_properties is a dict keyed by indexes of decision variables, mapping to 3-tuples: (var_type, lower_bound, upper_bound)
      • as long as the values in the lb array are always less or equal to the matching values in ub , their values are unrestricted floats;
      • var_type may assume 3 different values: 0 means binary, 1 means integer, 2 means continuous, refer to the constants VAR_TYPE_BINARY, VAR_TYPE_INTEGER, VAR_TYPE_CONTINUOUS in this file to use one of the supported variable types;
      • if var_type is set to VAR_TYPE_BINARY, the lower bounds must always be 0 and the upper bounds must always be 1 or an exception will be thrown;
      • any missing variables in vars_properties dict will be assumed to be integer ranging between [0, +∞).
    • if method is CLP, vars_properties is a dict keyed by indexes of decision variables, mapping to 2-tuples: (lower_bound, upper_bound);
      • as long as the lower bounds are always less or equal than upper bounds, their values are unrestricted floats;
      • all variables are assumed to be continuous;
      • any missing variables in vars_properties dict will be assumed to be continuous ranging between [0, +∞);
  • maximization: True if the LP model is to be maximized, False if it is to be minimized. Defaults to True;
  • method: a non-casesensitive string which defines which underneath solver structure provided by Google ORTools to be used:
    • BOP: binary integer programming, all decision variables belongs to {0, 1} (that is, they assume either 0 or 1) and the contents of vars_properties is ignored;
    • CBC: mixed integer programming, where variables may be binary, integer or continuous, as specified by vars_properties;
    • CLP: regular linear programming, where variables are always continuous and their interval is defined by vars_properties;
  • hint: a 1D python list of n elements to represent the non-optimal solution vector X, it will be used as a hint on a initial feasible solution for the solver to start on. This is a situational parameter, capable of speeding up the process of optimally solving the problem, it's only useful for specific problems which a initial feasible solution may be inferred without computationally demanding optimization techniques;
  • num_threads: the number of threads to be used on the optimization, default is 1. Note that there are platforms which doesn't support more than 1 thread;
Note that slack variables are created automatically when defining inequality constraints, therefore, it's not needed to worry about them if you are interested in delegating their creation and management. In addition, equality constraints may be created by setting the lower bound of a constraint to the exact same upper bound. Also, please, note there are two different types of lower and upper bounds: one of them is defined by the arrays lb and ub and the second is defined by the tuples present in vars_properties, however, they correspond to the bounds on the constraints and the bounds on each variable, respectively.

Outputs

The possible outputs are:
  • c*: the optimum objective value
  • X*: optimum solution as a 1D numpy array (in case of alternate optima this array is a single possible optimum solution);
  • status: the resulting status of the optimization, which could be: optimum, alternate optima, infeasible and unbounded;
  • time: the time needed to complete the optimization task in milliseconds;
  • iters: the number of iterations; demanded to find the optimum solution
  • bb_nodes: the number of branch and bound nodes created for solving an integer programming model;
The actual output depends on the choosen "method", based on what they support:
  • method BOP returns:
    • c*
    • X*
    • status
    • time
  • method CBC returns:
    • c*
    • X*
    • status
    • time
    • iters (number of iterations until the stopping condition is met)
    • bb_nodes (number of branch-and-bound nodes)
  • method CLP returns:
    • c*
    • X*
    • status
    • time
    • iters
def solve_lp(C, A, lb, ub, vars_properties={}, maximization=True, method='CBC', hint=[], num_threads=1):
	'''
	This function solves a given linear programming (LP) of the following type:
			max/min 	Cx
			s.t.		lb <= Ax <= ub
						vars_properties[i][j] <= x <= vars_properties[i][j+1]
						x belongs to {binary, integer, continuous}
	intput:
	------
		* C: the objective function to minimize or maximize as a 1D numpy array;
		* A: the constraints coefficients as a 2D numpy array;
		* lb: the lower bounds for each constraint as a 1D numpy array;
		* ub: the upper bounds for each constraint as a 1D numpy array;
		* vars_properties: a dictionary keyed by indexes of decision variables which defines the properties of each var as explained below:
			* if method is 'BOP', vars_properties won't be considered;
			* if method is 'CBC':
				* "vars_properties" is a dict keyed by indexes of decision variables, mapping to 3-tuples: (var_type, lower_bound, upper_bound)
				* as long as "lower_bound" is always less or equal to "upper_bound", their values are irrestricted floats
				* "var_type" may assyme 3 different values: 0 means binary, 1 means integer, 2 means continuous, refer to the constants "VAR_TYPE_BINARY", "VAR_TYPE_INTEGER", "VAR_TYPE_CONTINUOUS" in this file to use one of the supported variable types;
				* if "var_type" is set to "VAR_TYPE_BINARY", the "lower_bound" must always be 0 and the "upper_bound" must always be 1 or an exception will be thrown
				* any missing variables in vars_properties dict will be assumed to be integer ranging between [0, +infinity).
			* if method is 'CLP', vars_properties is a dict keyed by indexes of decision variables, mapping to 2-tuples: (lower_bound, upper_bound);
				* as long as "lower_bound" is always less or equal than "upper_bound", their values are unrestricted floats
				* all variables are assumed to be continuous;
				* any missing variables in "vars_properties" dict will be assumed to be continuous ranging between [0, +infinity);
		* maximization: True if the LP model is to be maximized, False if it is to be minimized;
		* method: a non-casesensitive string which defines which underneath solver structure provided by Google ORTools to be used:
			* 'BOP': binary integer programming, all decision variables belongs to {0, 1} and the contents of "vars_properties" is ignored;
			* 'CBC': mixed integer programming, where variables may be binary, integer or continuous, as specified by "vars_properties";
			* 'CLP': regular linear programming, where variables are always continuous and their interval is defined by "vars_properties";
		* num_threads: the number of threads to be used on the optimization, default is 1. Note that there are platforms which doesn't support more than 1 thread;
	output:
	------
		The possible outputs are:
			* C*: the optimum objective value
			* X*: optimum solution as a 1D numpy array (in case of alternate optima this array is a single possible optimum solution)
			* status: the resulting status of the optimization, which could be: optimum, alternate optima, infeasible and unbounded
			* time: the time needed to complete the optimization task in milliseconds
			* iters: the number of iterations demanded to find the optimum solution
			* bb_nodes: the number of branch and bound nodes created for solving an integer programming model

		The actual output depends on the choosen "method", based on what they support:
			* method 'BOP' returns:
				* C*
				* X*
				* status
				* time
			* method 'CBC' returns:
				* C*
				* X*
				* status
				* time
				* iters
				* bb_nodes
			* method 'CLP' returns:
				* C*
				* X*
				* status
				* time
				* iters
	Exceptions:
	----------
		Raises an exception of type "Exception", each with it's own customized message, due to model inconsistency problems, in the following cases:
			* the number of variables (length over axis 0) in C is different than the number of variables (columns) in A;
			* the number of constraints (length over axis 0) in A, lb and ub mismatch;
			* the number of variables (keys) in "vars_properties" is greater than the number of variables (columns in A) in the model;
			* if method is 'CBC' and the size of tuples in "vars_properties" is not exactly 3;
			* if method is 'CBC' and the first element in any 3-tuple in "vars_properties" is not VAR_TYPE_BINARY, VAR_TYPE_INTEGER or VAR_TYPE_CONTINUOUS;
			* if method is 'CBC' and the first element in any 3-tuple in "vars_properties" is VAR_TYPE_BINARY but the lower_bound is not 0 or the upper_bound is not 1;
			* if method is 'CLP' and the size of tuples in "vars_properties" is not exactly 2;
			* if method is 'CBC' or 'CLP' and for any uple found in "vars_properties" the lower_bound is strictly greater than the upper_bound
			* independently of method, if the k-th item in lb is strictly greater than the corresponding k-th item in ub for any valid k

	'''
	if C.shape[0] != A.shape[1]:
		raise Exception('The number of variables in C and A doesn\'t match.')
	if (A.shape[0] != lb.shape[0]) or (A.shape[0] != ub.shape[0]):
		raise Exception('The number of variables in lb, ub and A doesn\'t match.')

	n = A.shape[1]
	c = A.shape[0]
	X = []

	# determine the solver method
	solver_method = None
	solver_name = ''
	if method.upper() == 'BOP':
		solver_method = pywraplp.Solver.BOP_INTEGER_PROGRAMMING
		solver_name = 'binary_integer_programming'
	elif method.upper() == 'CBC':
		if len(vars_properties.keys()) > A.shape[1]:
			raise Exception('Too many variables properties for too few decision variables in the model.')
		solver_method = pywraplp.Solver.CBC_MIXED_INTEGER_PROGRAMMING
		solver_name = 'mixed_integer_programming'
	elif method.upper() == 'CLP':
		if len(vars_properties.keys()) > A.shape[1]:
			raise Exception('Too many variables properties for too few decision variables in the model.')
		solver_method = pywraplp.Solver.CLP_LINEAR_PROGRAMMING
		solver_name = 'continuous_linear_programming'
	else:
		raise Exception('Unsupported method:', method)
	# creates the solver instance
	solver = pywraplp.Solver(solver_name, solver_method)
	infinity = solver.infinity()

	# sets the number of threads (currently doesn't works for some platforms like windows)
	if num_threads <= 0:
		num_threads = 1
	else:
		num_threads = int(num_threads)
	solver.SetNumThreads(num_theads=num_threads)

	# creates the variables, setting all their properties
	if method.upper() == 'BOP':
		for k in range(0, n):
			variable = solver.BoolVar('x[%i]' % k)
			X.append(variable)

	elif method.upper() == 'CBC':
		for k in range(0, n):
			kth_type = 1
			kth_lb = 0.0
			kth_ub = infinity
			if k in vars_properties:
				try:
					kth_type, kth_lb, kth_ub = vars_properties[k]
				except ValueError:
					raise Exception('The "CBC: method expected tuples of size exactly 3 in the "vars_properties" dictionary.')
			if kth_lb > kth_ub:
				raise Exception('Inconsistent '+str(k)+'-th variable bounds: the lower bound "'+str(kth_lb)+'" is greater than the upper bound "'+str(kth_ub)+'".')

			variable = None
			if kth_type == 0:
				if kth_lb != 0 or kth_ub != 1:
					raise Exception('Binary variables must always have their lower bound set to 0 and their upper bound set to 1.')
				variable = solver.BoolVar('x[%i]' % k)
			elif kth_type == 1:
				variable = solver.IntVar(kth_lb, kth_ub, 'x[%i]' % k)
			elif kth_type == 2:
				variable = solver.Var(kth_lb, kth_ub, False,'x[%i]' % k)
			else:
				raise Exception('Unknown variable type in vars_properties:', kth_type)
			X.append(variable)

	elif method.upper() == 'CLP':
		for k in range(0, n):
			kth_lb = 0.0
			kth_ub = infinity
			if k in vars_properties:
				try:
					kth_lb, kth_ub = vars_properties[k]
				except ValueError:
					raise Exception('The "CLP: method expected tuples of size exactly 2 in the "vars_properties" dictionary.')
			if kth_lb > kth_ub:
				raise Exception('Inconsistent '+str(k)+'-th variable bounds: the lower bound "'+str(kth_lb)+'" is greater than the upper bound "'+str(kth_ub)+'".')

			variable = solver.Var(kth_lb, kth_ub, False,'x[%i]' % k)
			X.append(variable)

	# creates the constraints, setting all their properties
	for k in range(0, c):
		kth_lb = lb.item(k)
		kth_ub = ub.item(k)
		if kth_lb > kth_ub:
			raise Exception('Inconsistent '+str(k)+'-th constraint bounds: the lower bound "'+str(kth_lb)+'" is greater than the upper bound "'+str(kth_ub)+'".')

		constraint = solver.RowConstraint(kth_lb, kth_ub, 'constraint[%i]' % k)
		for v in range(0, n):
			constraint.SetCoefficient(X[v], A.item(k, v))

	# sets the LP objective
	objective = solver.Objective()
	for k in range(n):
		objective.SetCoefficient(X[k], C.item(k))
	if maximization:
		objective.SetMaximization()
	else:
		objective.SetMinimization()

	# sets a hint for a initial basic feasible solution
	if len(hint) > 0:
		if len(hint) != n:
			raise Exception('The hint list has an inconsistent number of elements:', len(hint))
		solver.SetHint(solver.variables(), hint)

	# attempts to solve the problem
	status = solver.Solve()
	if method.upper() == 'BOP':
		return solver.Objective().Value(), np.array([variable.solution_value() for variable in X]), status, solver.wall_time()
	elif method.upper() == 'CBC':
		return solver.Objective().Value(), np.array([variable.solution_value() for variable in X]), status, solver.wall_time(), solver.iterations(), solver.nodes()
	elif method.upper() == 'CLP':
		return solver.Objective().Value(), np.array([variable.solution_value() for variable in X]), status, solver.wall_time(), solver.iterations()
There are mathematical and software consistencies which must be obeyed for the above defined function.

Exceptions

Raises an exception of type Exception, each with it's own customized message, due to model inconsistency problems, in the following cases:
  • the number of variables (length over axis 0) in C is different than the number of variables (columns) in A;
  • the number of constraints (length over axis 0) in A, lb and ub mismatch;
  • the number of variables (keys) in vars_properties is greater than the number of variables (columns in A) in the model;
  • if method is CBC and the size of tuples in vars_properties is not exactly 3;
  • if method is CBC and the first element in any 3-tuple in vars_properties is not VAR_TYPE_BINARY, VAR_TYPE_INTEGER or VAR_TYPE_CONTINUOUS;
  • if method is CBC and the first element in any 3-tuple in vars_properties is VAR_TYPE_BINARY but the lower bound is not 0 or the upper bound is not 1;
  • if method is CLP and the size of tuples in vars_properties is not exactly 2;
  • if method is CBC or CLP and for any tuple found in vars_properties the lower bound is strictly greater than the upper bound;
  • independently of method, if the k-th item in lb is strictly greater than the corresponding k-th item in ub for any valid k;

Conclusion

Fortunately LP is a powerful mathematical tool to solve real world problems which requires optimization to be employed, however, only the (fully) continuous LP formulations belongs to the Polynomial class of problems. In other words, if a LP problem admits the optimum solution vector X* to have all of it's entries as continuous real-valued numbers such problem have computational complexity belonging to the Polynomial class and it may be solved in feasible time by computer machines. Unfortunately, since MIP problems constrains the optimum solution X* to have at least one of it's entries as integer-valued number(s), the computational complexity class to which such formulation belongs is the Non-Deterministic Polynomial Hard (NP-Hard) class, which requires an explosive amount of operations to solve arbitrary problems. Further, MIP problems may only be solved in feasible time for small instances or particular large instances. That's because, for some NP-Complete and NP-Hard problems, particular cases of them may have been proven to belong to the Polynomial class, while the most generic form of such problems are still NP-Complete or NP-Hard.

It makes perfect sense that there are MIP problems belonging to the NP-Complete and NP-Hard classes, because it's such a powerful technique, even though it's based only on linear constraints and objective function, that it's possible to model graph coloring, for an example. Graph coloring is one of Karp's 21 NP-Complete problems, therefore, even nowadays there is no known technique capable of solving the most generic form of such problem in polynomial time, no matter if it's been formulated as a MIP problem or any other formulation/algorithm.
